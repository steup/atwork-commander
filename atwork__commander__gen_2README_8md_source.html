1 # atwork\_commander\_gen
2 
3 Enable dynamic task generation according to specified task types of the [@Work
4 RuleBook](https://github.com/robocup-at-work/rulebook).
5 
6 ## Architecture
7 
8 TaskGenerator with pluggable generators and configuration infrastructure.
9 
10 The ```TaskGenerator``` class if the interface to embed the functionality in other programs. The class uses
11 ros-parameters to extract configuration information including the Arena and Tasks-Specification as well as the genrator
12 plugin to be used.
13 
14 The generator plugins are implemented using ROS pluginlib. The implementation currently uses pluginlib, logging and
15 the parameter-server API of ROS.
16 
17 The ```TaskGenerator``` is no ROS-Node, but a library to be used in other ROS-Nodes like the refbox state-machine from
18 the ```atwork_commander_core```.

38 ## Configuration
39 
40 Uses simple ROS parameter-based specification of the task types and the arena, which is used to generate task
41 instances accordingly. The general approach is to specify the configuration using ```.yaml``` files, which are either
42 manually loaded using ```rosparam load``` or automatically loaded within ```.launch``` files.
43 
44 ### Arena Configuration
45 
46 The arena, where the task are executed in, is specified as a hierarchical parameter in the parameter
47 server. The name of the this parameter is passed on ```TaskGenerator``` construction through the ```arenaConfig```
48 parameter. The arena parameter contains the following sub-parts.
49 
50 By default the parameter ```arena``` is used to read the arena specification from in the launch files
51 ```atwork_commander atwork_commander.launch``` and ```atwork_commander_gen example.launch```.
52 
53 An example arena configuration can be found in ```atwork_commander_gen/config/example-arena.yaml```
54 
55 #### Workstations
56 
57 The ```workstations``` sub-parameter contains individual key-value pairs of workstation names and the according
58 workstation type ```<ws-name>: "<ws-type>"```. An example Workstation specification looks like this:
59 
60 ```
61 workstations:
62  WS01: "10"
63  WS02: "15"
64  WS03: "05"
65  WS04: "00"
66  WS06: "15"
67  WS08: "05"
68  PP01: "PP"
69  TT01: "TT"
70  SH02: "SH"
71 ```

89 #### Cavities
90 
91 Available cavities in the arena are specified as a list of cavity names e.g.:
92 
93 ```
94 cavities: [ "F20_20_H", "S40_40_H", "S40_40_V", "M20_H", "M20_V", "M30_H", "M30_V", "M20_100_H", "R20_H", "R20_V"
95 ```
96 
97 Each cavitiy description consists of the form part e.g. ```F20_20``` for small aluminium profile and a specifier for
98 the orientation. The orientation may either be ```_H``` for horizontal placement or ```_V``` for vertical placement.
99 The reference face for this placement is always the face with the largest area.
100

104

106 key-value pairs of ```<object-type>: <count>``` entries in the sub-parameter ```objects```.
107 
108 The idea is to enable the task generator to only use up as many objects of each type as available

132 ### Task Configuration
133 
134 Task configuration consists of multiple sub-specifications indicating different aspects of the task. The different
135 configuration options will be sorted based on their data type.
136 
137 All configurations options are sub-parameters of a task. Each task uses the task name as thesub-parameter name. The
138 ```TaskGenerator```'s second options taskConfig describes the parameter, which contains the individual task types as
139 sub-parameters. Typically this parameter is ```tasks```.
140 
141 The possible sub-parameters of each task have a default parameter, which is used if the parameter is not specified. If
142 parameters are specified that are unknown to the config parser a warning will be printed, but no error is raised.
143 
144 The default values are specified in `atwork_commander_gen/include/atwork_commander_gen/ConfigParserInterface.h`
145 
146 #### Integer Configurations
147 
148 Most of the configuration parameters of each task are integer variables. Boolean choices are also represented through
149 integer variables. In the following there are all the currently supported integer variables described.
150 
151 Name | Range | Default | Description
152 -----------------------|----------------|---------|------------
153 prep_time | 1 - 2147483647 | 1 | Time in minutes for the preparation phase
154 exec_time | 1 - 2147483647 | 1 | Time in minutes for the execution phase
155 waypoints | 0 - 2147483647 | 0 | Number of waypoints to generate (Not implemented for JurekGen)
156 objects | 0 - 2147483647 | 1 | Number of objects to transport
157 decoys | 0 - 2147483647 | 0 | Number of decoys to place on workstations
158 barrier_tapes | 0 - 2147483647 | 0 | Number of barrier tapes to be placed in the arena (Not yet implemented)
159 obstacles | 0 - 2147483647 | 0 | Number of obstacles to be placed in the arena (Not yet implemented)
160 arbitrary_surfaces | 0 - 2147483647 | 0 | number of workstations, where objects are picked from, which contain arbitrary surfaces (not yet implemented)
161 tables | 1 - 2147483647 | 0 | number of workstations to be used in task (not yet implemented for JurekGen)
162 tables |
163 pp_placing | 0 - 2147483647 | 0 | Number of objects to be placed on a PPT in the appropriate cavity
164 ref_cavity_position | 0 - 1 | 0 | Boolean if cavity position is decided by referees (Ignored, currently assumed to be always 1)
165 ref_cavity_rotation | 0 - 1 | 0 | Boolean if cavity rotation is decided by referees (Not yet implemented)
166 ref_cavity_orientation | 0 - 1 | 0 | Boolean if cavity orientation is decided by referees (Not implemented in JurekGen)
167 container_placing | 0 - 2147483647 | 0 | How many objects to be placed in containers on workstations
168 tt_grasping | 0 - 2147483647 | 0 | How many objects to be grapsed from RTTs and CBTs
169 tt_placing | 0 - 2147483647 | 0 | How many objects to be placed on RTTs or CBTs
170 shelfes_grasping | 0 - 2147483647 | 0 | How many objects to be grasped from Shelves
171 shelfes_placing | 0 - 2147483647 | 0 | How many objects to be placed on Shelves
172 container_on_shelf | 0 - 1 | 0 | If containers are allowed to be placed on shelfes
173 container_on_tt | 0 - 1 | 0 | If containers are allowed ot be placed on RTTs and CBTs
174 ref_tt_direction | 0 - 1 | 0 | If the direction of the RTT is set by the referees (Not implemented
175 yet)
